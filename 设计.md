下面给你一份基于 **.NET + GtkSharp** 的设计文档，默认面向 **Ubuntu 桌面环境**，目标是做一个 **本地文件夹壁纸管理 + 自动切换** 的桌面应用。文档风格偏工程实现，可直接作为开发设计说明书使用。

# Ubuntu 壁纸管理软件设计文档

## 1. 文档信息

**项目名称**：WallpaperUbuntu
**技术栈**：.NET + GtkSharp
**运行平台**：Ubuntu 24.04 作为测试平台优先满足
**兼容平台**: Ubuntu 20.04+, 兼容其他 GNOME 桌面 Linux 发行版
**文档目的**：定义系统架构、模块划分、核心流程、数据结构与关键实现方案，指导开发落地。

---

# 2. 设计目标

本项目是一个运行在 Ubuntu 上的轻量级壁纸管理软件，支持从用户选择的本地文件夹中读取图片，并按照设定规则自动切换壁纸。

## 2.1 核心目标

* 支持选择一个或多个本地壁纸文件夹
* 支持定时自动切换壁纸
* 支持顺序、随机、不重复随机等策略
* 提供 GtkSharp 图形界面进行配置和控制
* 在 GNOME 环境下可靠设置桌面壁纸
* 支持开机自启动
* 尽量降低资源占用，长期后台稳定运行

## 2.2 非目标

当前版本暂不重点支持：

* 在线壁纸下载
* 云同步
* 多桌面环境深度适配
* 多显示器独立壁纸策略
* 图像编辑能力

# 4. 技术选型说明

## 4.1 .NET

建议使用：

* **.NET 8**
* 原因：

  * LTS，稳定
  * Linux 支持成熟
  * 性能较好
  * 生态足够完善

## 4.2 GtkSharp

建议使用 GtkSharp 构建原生 Linux GUI。

优点：

* 与 Ubuntu 桌面风格兼容较好
* 原生 GTK 体验优于跨平台模拟控件
* 适合做轻量桌面工具

## 4.3 壁纸设置方式

在 Ubuntu GNOME 环境下优先通过命令或 GSettings 设置壁纸：

* `org.gnome.desktop.background picture-uri`
* `org.gnome.desktop.background picture-uri-dark`

建议封装为统一接口，由 `GnomeWallpaperService` 实现。

### 5.2.1 WallpaperAppService

应用核心服务，负责协调 UI、调度器、配置和壁纸切换。

#### 主要职责

* 启动应用时加载配置
* 初始化壁纸目录和图片缓存
* 响应 UI 操作
* 调用壁纸切换逻辑
* 管理运行状态
* 在设置变更后保存配置

#### 关键方法

```csharp
Task InitializeAsync();
Task NextWallpaperAsync();
Task PreviousWallpaperAsync();
Task StartAutoSwitchAsync();
Task PauseAutoSwitchAsync();
Task RefreshCatalogAsync();
Task SaveSettingsAsync();
```

---

### 5.2.2 SchedulerService

负责定时切换任务。

#### 设计要求

* 支持启动
* 支持暂停
* 支持恢复
* 支持动态更新切换间隔
* 避免重入执行

#### 建议实现

使用 `System.Threading.Timer` 或 `PeriodicTimer`。

建议优先：

* `PeriodicTimer` + 后台 Task
* 原因：逻辑更直观，便于 async/await 管理

#### 核心状态

* 是否运行中
* 当前周期
* 上次执行时间
* 当前任务是否正在执行

### 5.4.2 GnomeWallpaperService

负责与 Ubuntu GNOME 交互设置壁纸。

#### 接口建议

```csharp
public interface IWallpaperSystemService
{
    Task SetWallpaperAsync(string filePath);
    Task<string?> GetCurrentWallpaperAsync();
}
```

#### GNOME 实现

可通过以下方式之一：

### 方案 A：调用 `gsettings`

优点：

* 实现简单
* 调试方便

缺点：

* 依赖系统命令
* 错误处理需要额外封装

### 方案 B：通过 Gio/GSettings 绑定

优点：

* 更原生
* 避免命令行依赖

缺点：

* GtkSharp / GI 绑定实现复杂度可能更高

#### 当前建议

V1 使用 **命令行调用 gsettings**，后续可抽象替换。

#### 示例逻辑

```bash
gsettings set org.gnome.desktop.background picture-uri "file:///home/user/Pictures/a.jpg"
gsettings set org.gnome.desktop.background picture-uri-dark "file:///home/user/Pictures/a.jpg"
```

#### 需要注意

* 路径必须转为 URI
* 对空格与特殊字符要正确转义
* 文件不存在时要提前校验

# 6. 核心业务流程

---

## 6.1 启动流程

```text
应用启动
  -> 加载配置
  -> 初始化日志
  -> 扫描壁纸目录
  -> 生成壁纸列表
  -> 初始化策略对象
  -> 加载主窗口
  -> 若启用自动切换，则启动 Scheduler
```

---

## 6.2 手动切换下一张

```text
用户点击“下一张”
  -> WallpaperAppService.NextWallpaperAsync()
  -> 当前策略计算下一张
  -> 调用 GnomeWallpaperService.SetWallpaperAsync()
  -> 更新 AppState
  -> 更新 UI 显示
  -> 写入日志
```

---

## 6.3 自动切换流程

```text
Scheduler 周期触发
  -> 检查是否暂停
  -> 检查当前是否已有切换任务在执行
  -> 计算下一张壁纸
  -> 设置系统壁纸
  -> 更新状态
  -> 刷新 UI（若窗口可见）
```

---

## 6.4 更新设置流程

```text
用户修改设置
  -> UI 校验输入
  -> 更新 AppConfig
  -> 保存配置
  -> 如目录或递归选项改变，则重新扫描目录
  -> 如间隔改变，则重启 Scheduler
  -> 如模式改变，则切换 Strategy
  -> 如开机启动改变，则更新 autostart
```

# 8. 并发与线程模型

GtkSharp UI 更新必须发生在主线程。

因此设计上应区分：

* UI 线程：窗口交互、界面刷新
* 后台线程：文件扫描、定时任务、系统命令调用

## 8.1 原则

* 文件扫描放后台线程
* 壁纸切换可在后台执行
* 切换完成后通过 GTK 主线程调度更新 UI

## 8.2 建议方式

使用：

* `Task.Run()` 处理 IO 密集任务
* `PeriodicTimer` 负责调度
* `GLib.Idle.Add` 或 GTK 对应主线程调度机制刷新界面

# 10. 错误处理设计

---

## 10.1 常见错误场景

### 文件夹不存在

处理：

* UI 提示
* 在扫描结果中跳过
* 日志记录

### 文件夹无权限

处理：

* 跳过该目录
* 日志告警

### 图片格式不支持

处理：

* 扫描阶段忽略

### 壁纸设置失败

处理：

* 保持当前状态不更新
* 提示“设置壁纸失败”
* 记录底层错误

### 配置文件损坏

处理：

* 备份原文件
* 恢复默认配置
* 提示用户配置已重置

### 定时器重复触发

处理：

* 增加执行锁
* 若上次未完成，忽略本次触发
